# 医疗领域知识(HIS / EMR)

> 本文档梳理 his-platform 涉及的医疗信息化领域知识:核心概念、业务流程、行业标准。既是项目的领域背景,也是面试时讲清"为什么这么建模"的底稿。
>
> **项目边界**:聚焦门诊主线,展示领域建模与标准对接。**不**死磕国家互联互通测评、真实医保接口、住院全流程、LIS/PACS 对接。

---

## 一、HIS 与 EMR 是什么

| 系统 | 全称 | 定位 |
|------|------|------|
| **HIS** | Hospital Information System 医院信息系统 | 覆盖医院**管理与业务流程**:挂号、就诊、医嘱、收费、药房、床位等,偏"流程与钱" |
| **EMR** | Electronic Medical Record 电子病历 | 患者**临床医疗文档**:病历、诊断、处方、检验检查报告等,偏"临床数据" |
| **EHR** | Electronic Health Record 电子健康档案 | 跨机构的、贯穿一生的健康记录(比 EMR 范围更大,本项目不涉及) |

关系:HIS 是骨架(业务流转),EMR 是血肉(临床内容)。一次就诊在 HIS 里走完挂号→医嘱→收费的流程,同时在 EMR 里沉淀出病历文档。

## 二、核心概念术语表

### 患者主索引 EMPI
- **EMPI**(Enterprise Master Patient Index,企业级患者主索引):在多系统/多院区中唯一标识"同一个自然人"的主数据。
- 解决"同一个人在不同科室/系统建了多份档案"的问题,是所有临床数据挂靠的锚点。
- 本项目:`his-mdm` 维护患者主索引,建档时 `gender` 必填(MALE/FEMALE/UNKNOWN),下游一律引用患者 id。

### 挂号 · 排班 · 号源
- **排班(Schedule)**:医生在某日某时段(上午 AM / 下午 PM)出诊的计划,决定可放多少号。
- **号源(Slot)**:排班下的可预约名额,是稀缺资源 —— **超挂(超卖)是医疗信息化的经典并发问题**。
- **挂号(Appointment/Registration)**:患者占用一个号源,产生就诊资格。
- 本项目:`his-registration` 用 **Redis Lua 原子锁**扣减号源,防止高并发超挂;`period` 必填(AM/PM)。

### 就诊 Encounter
- **Encounter(就诊/接触)**:患者与医疗机构的一次医疗接触(一次门诊、一次住院都是一个 Encounter)。是医嘱、诊断、病历、账单挂靠的核心聚合根。
- 本项目 `EncounterStatus`:`OPEN`(就诊中)→ `SUBMITTED`(医嘱已提交)→ `CLOSED`(结束)。
- **`encounterId` 是全局幂等键**:账单、病历都按它 upsert,保证"一次就诊只有一份账单/病历"。

### 医嘱 Order ⭐
- **医嘱(Medical Order)**:医生下达的诊疗指令,是门诊/住院的核心业务对象。
- **医嘱类型**(本项目 `OrderType`):
  - `DRUG` 药品医嘱(处方)—— **需药师审方**
  - `EXAM` 检查医嘱(如影像、心电)
  - `TREATMENT` 治疗医嘱(如输液、换药)
- 现实中还有检验(LAB)、手术、护理、膳食等,本项目取三类展示。

### 处方审核(审方)
- **审方(Prescription Review)**:药师对药品医嘱做**合理性审核**(配伍禁忌、剂量、重复用药、相互作用等),是**用药安全**的关键关卡,也是《医疗机构处方审核规范》的要求。
- 只有药品医嘱进审方队列;审核**通过**才可执行、才计费,**驳回**需医生修改后重提交。
- 本项目:`his-outpatient` 的 `PrescriptionReviewService` + 状态机 `PENDING_REVIEW`/`REJECTED`,药师角色(PHARMACIST)操作,留痕。

### 收费 · 结算 · 退费
- **计费/结算(Billing)**:根据医嘱项目生成费用账单;门诊多为"先划价后收费"。
- **缴费(Payment)**:患者支付账单。本项目缴费后才推进医嘱执行 + 生成病历(先付费后服务的门诊模型)。
- **退费(Refund)**:已缴费项目的撤销退款,通常需**审批**(涉及资金,防止随意退)并联动**撤销已执行的医嘱**。
- 本项目:`his-billing` 事件驱动计费;退费审批状态机 `REQUESTED → APPROVED/REJECTED`,批准后联动 outpatient 撤销医嘱。

### 电子病历文档 EMR
- 一次就诊沉淀的临床文书:主诉、现病史、诊断、处方、医嘱等结构化/半结构化文档。
- 本项目:`his-emr` 在缴费后自动生成病历文档,按 `encounterId` 幂等,并对外提供 FHIR 标准视图。

## 三、门诊业务全流程(端到端)

```
① 建档(EMPI)         his-mdm      患者主索引
      │
② 挂号(占号源)        his-registration  Redis 锁防超挂
      │
③ 就诊 Encounter      his-outpatient    医生开单、下诊断
      │
④ 开医嘱 Order        his-outpatient    按类型路由
      │
      ├─ 药品 DRUG ──▶ ⑤ 药师审方 ──通过──┐
      │                        └─驳回─▶ 医生重提交
      └─ 检查/治疗 ───────────────────────┤
                                          ▼
⑥ 计费(事件 order.placed)  his-billing   生成账单
      │
⑦ 缴费 pay(事件 invoice.paid)          推进医嘱执行
      │                                   ├─▶ 医嘱 EXECUTED
      │                                   └─▶ his-emr 生成病历(FHIR)
      │
⑧ 退费(可选,审批流)  his-billing        批准 → 撤销已执行医嘱
      │
各节点 ──事件──▶ his-notify 落地待办/通知(审方待办、退费待办、结果通知)
```

**关键医疗语义**:审方通过前不计费(避免驳回后要退费);缴费后才执行(先付费后服务);退费必须审批且联动撤销医嘱(资金与临床一致)。

## 四、医嘱状态机(核心领域模型)⭐

医嘱是**充血领域模型 + Spring StateMachine**。状态与事件:

**状态(OrderState)**:`CREATED` → `PENDING_REVIEW` / `REJECTED` → `SUBMITTED` → `EXECUTED` → `STOPPED` / `CANCELLED`

**事件(OrderEvent)**:`SUBMIT` / `SUBMIT_FOR_REVIEW` / `REVIEW_PASS` / `REVIEW_REJECT` / `RESUBMIT` / `EXECUTE` / `STOP` / `CANCEL` / `REFUND`

```
                    非药品(EXAM/TREATMENT)
 CREATED ──SUBMIT──────────────────────────▶ SUBMITTED ──EXECUTE──▶ EXECUTED
   │                                            │  │                   │
   │  药品(DRUG)                                │  └──STOP──▶ STOPPED   │
   ├─SUBMIT_FOR_REVIEW─▶ PENDING_REVIEW         │                      │
   │                        │  │                CANCEL                REFUND
   │           REVIEW_PASS  │  │ REVIEW_REJECT     │                    │
   │              ▼         │  ▼                   ▼                    ▼
   │          SUBMITTED     │ REJECTED         CANCELLED            CANCELLED
   │              ▲         │  │                                  (退费撤销)
   └──CANCEL──▶ CANCELLED   └──┘ RESUBMIT ▶ PENDING_REVIEW
```

- **审方分叉**:药品走 `SUBMIT_FOR_REVIEW → PENDING_REVIEW`,药师 `REVIEW_PASS` 进 `SUBMITTED`,`REVIEW_REJECT` 进 `REJECTED`,医生 `RESUBMIT` 回审方。
- **执行**:缴费事件驱动 `EXECUTE`(SUBMITTED → EXECUTED)。
- **撤销**:未执行可 `CANCEL`;已执行医嘱经**退费**事件 `REFUND` 置 `CANCELLED`。
- 用状态机而非 if-else,把"非法流转"(如已执行又去审方)交给状态机守卫拦截,领域规则集中且可测。

## 五、行业标准:FHIR / HL7

### HL7 与 FHIR
- **HL7**(Health Level Seven):医疗信息交换标准组织,V2/V3 是早期消息标准。
- **FHIR**(Fast Healthcare Interoperability Resources,读作 "fire"):HL7 推出的**现代化医疗互操作标准**,基于 **REST + JSON/XML + 资源(Resource)模型**,是当前医疗数据交换的事实标准。**R4** 是广泛采用的稳定版本。
- 特点:一切皆"资源",资源有标准字段和引用关系,通过 RESTful API 读写,天然适配微服务。

### 本项目用到的 FHIR R4 资源

| 资源 | 含义 | 对应本系统 |
|------|------|-----------|
| **Patient** | 患者人口学信息 | 患者主索引 EMPI |
| **Encounter** | 一次医疗接触 | 就诊记录 |
| **Composition** | 临床文档(病历)的结构骨架 | 病历文档 |
| **Condition** | 诊断/健康问题 | 就诊诊断 |
| **MedicationRequest** | 用药医嘱/处方 | 药品医嘱 |

- 本项目 `his-emr` 用 **HAPI FHIR**(FHIR 的 Java 参考实现)把内部模型映射成标准资源,以 `application/fhir+json` 对外输出(`GET /fhir/Patient/{id}`、`GET /fhir/Encounter/{id}`)。
- 现实还有 `Observation`(检验检查结果)、`Practitioner`(医务人员)、`MedicationDispense`(发药)、`DiagnosticReport`(诊断报告)等,本项目取核心几类展示标准对接能力。

### 为什么标准很重要
医院内有几十套异构系统(HIS/LIS/PACS/EMR…),FHIR 提供统一的资源模型和 API,是"互联互通"(区域医疗、分级诊疗、跨院调阅)的技术基础。国内还有基于 HL7/IHE 的"电子病历互联互通测评"体系(本项目不做测评认证,只做标准输出演示)。

## 六、角色与权限(临床岗位)

医疗系统的权限强绑定**岗位职责**,本项目 RBAC 角色:

| 角色 | 职责 | 关键权限(示例) |
|------|------|---------|
| **医生 DOCTOR** | 接诊、下诊断、开医嘱 | 就诊、开/提交/撤销医嘱、写病历 |
| **护士 NURSE** | 建档、协助挂号、执行医嘱 | 患者建档、挂号 |
| **收费员 CASHIER** | 划价、收费、发起退费 | 缴费、发起退费申请 |
| **药师 PHARMACIST** | 处方审核、发药 | 审方通过/驳回 |
| **管理员 ADMIN** | 系统与流程管理、审批 | 排班、退费审批 |

**职责分离(SoD)**:发起退费的是收费员、审批的是管理员;开药的是医生、审方的是药师 —— 关键操作两人两岗,是医疗合规与风控的基本要求,也是本项目审批流的设计动机。

## 七、医疗信息化的几个"隐形难点"(本项目如何呼应)

| 难点 | 医疗语义 | 本项目对应 |
|------|---------|-----------|
| **号源不能超卖** | 一个时段就那么多号,超挂影响诊疗秩序 | Redis Lua 原子锁 |
| **用药安全** | 错误处方可能致命,必须审方 | 药师审方状态机 + 计费推迟 |
| **先付费后服务** | 门诊多为缴费后执行 | 缴费事件驱动医嘱执行 |
| **资金一致性** | 退费要连带撤销临床动作,不能钱退了医嘱还在 | 退费审批 + 事件联动撤销医嘱 |
| **一次就诊一份账单/病历** | 幂等,避免重复计费/重复建档 | `encounterId` 幂等键 + upsert |
| **数据可追溯** | 医疗数据需审计留痕 | `AuditableEntity` who/when + 审方/退费留痕 |
| **系统互联互通** | 异构系统需标准交换 | FHIR R4 标准输出 |
| **隐私与最小权限** | 患者数据敏感,按岗位授权 | RBAC + 科室数据权限 |
